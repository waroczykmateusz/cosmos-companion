# Cosmic Companion — Plan MVP (Flutter, offline-first)

## Context

Budujemy nową aplikację mobilną Flutter dla pasjonatów astrofotografii i astrologii. Cel: hybrydowe doświadczenie łączące precyzyjne dane naukowe (pozycje ciał niebieskich, widoczność, fazy księżyca) z warstwą symboliczną (znaki zodiaku, domy, aspekty, tranzyty). Aplikacja musi działać offline (sesje w terenie, daleko od miast), z UI zoptymalizowanym pod ciemne niebo (Night Vision First, Red Mode).

Repo: cosmic-companion (GitHub), branch roboczy cosmic-companion, merge do master przy gotowych funkcjach.

Pierwszy kamień milowy: Dashboard MVP "Gdańsk Center" — lokalizacja GPS, faza Księżyca, znak zodiaku Księżyca, mock seeing, Night Mode toggle.

## Decyzje architektoniczne (potwierdzone z użytkownikiem)

| Obszar          | Wybór                                            | Uzasadnienie |
|-----------------|--------------------------------------------------|--------------|
| Astro engine    | Swiss Ephemeris (FFI) + astronomy_engine_dart    | SwissEph: profesjonalna astrologia (domy Placidus/Koch, aspekty, tranzyty). astronomy_engine: szybkie pozycje ciał + widoczność/risetimes. |
| State mgmt      | Riverpod 2.x                                     | Mniej boilerplate niż bloc, type-safe DI, reaktywne providers idealnie pasują do strumieni czasu (live update pozycji ciał). |
| DB              | Drift (sqlite)                                   | Aktywnie rozwijany, type-safe SQL, reaktywne queries (watchSingle), bezpieczne migracje. |
| i18n            | PL + EN od MVP                                   | flutter_localizations + intl + .arb files od dnia 1. |
| Mapy            | maplibre_gl                                      | Light Pollution Map z custom tile sourcem (Bortle/VIIRS data). |
| Modele          | freezed + json_serializable                      | Immutable data classes, copyWith, equality. |
| Testy           | flutter_test + mocktail                          | Standard Flutter, brak code-genu dla mocków. |
| Auth (MVP)      | Local profile + PIN/biometria                    | Offline-first. Pluggable AuthProvider interface — w MVP LocalAuthProvider, potem swap na SupabaseAuthProvider bez refaktoru kodu konsumentów. |
| Secure storage  | flutter_secure_storage                           | Keystore (Android) / Keychain (iOS). NIGDY shared_preferences dla wrażliwych danych. |
| Compliance      | GDPR-ready od MVP                                | Telemetria opt-in, hard purge function, no PII in logs. Privacy policy stub w settings. |

## Model hybrydowy: CelestialBody

Plik: lib/data/models/celestial_body.dart

```dart
import 'package:freezed_annotation/freezed_annotation.dart';

part 'celestial_body.freezed.dart';
part 'celestial_body.g.dart';

/// Hybrydowy model ciała niebieskiego — łączy dane astronomiczne (pozycja
/// na sferze niebieskiej, jasność) z warstwą astrologiczną (znak zodiaku,
/// dom horoskopowy, aspekty względem innych ciał).
///
/// Wszystkie pola pozycyjne są snapshotem dla konkretnego momentu (epoch)
/// i obserwatora (observerLocation). Do długoterminowych obliczeń używaj
/// EphemerisService.computeAt(epoch, location).
@freezed
class CelestialBody with _$CelestialBody {
  const factory CelestialBody({
    required CelestialBodyId id,            // SUN, MOON, MERCURY, ..., PLUTO, NORTH_NODE, CHIRON
    required String displayName,             // "Księżyc" / "Moon" — z l10n
    required DateTime epoch,                 // UTC, dla którego liczono pozycję
    required GeoLocation observerLocation,   // lat/lon/elev obserwatora

    // === ASTRONOMIA (równikowy układ J2000) ===
    required double rightAscension,          // [h] 0..24, rektascensja
    required double declination,              // [°] -90..90, deklinacja
    required double distanceAU,              // [AU] odległość od Ziemi
    required double apparentMagnitude,       // jasność wizualna (Słońce: -26.7, Wenus max: -4.6)
    required double altitude,                 // [°] wysokość nad horyzontem (po refrakcji)
    required double azimuth,                  // [°] 0=N, 90=E, 180=S, 270=W
    required bool isAboveHorizon,
    DateTime? riseTime,                       // null = ciało okołobiegunowe lub niewidoczne
    DateTime? transitTime,                    // moment kulminacji
    DateTime? setTime,

    // === ASTROLOGIA (układ ekliptyczny, geocentryczny) ===
    required double eclipticLongitude,        // [°] 0..360, długość ekliptyczna
    required double eclipticLatitude,         // [°] szerokość ekliptyczna
    required ZodiacSign zodiacSign,           // znak na podstawie eclipticLongitude
    required double signDegree,               // [°] 0..30, pozycja w obrębie znaku
    required AstrologicalHouse? house,        // dom 1-12, null gdy nie liczymy karty
    required bool isRetrograde,               // ruch wsteczny
    @Default([]) List<Aspect> aspects,        // aspekty do innych ciał (cieżkie tylko: koniunkcja/opozycja/kwadratura/trygon/sekstyl)
  }) = _CelestialBody;

  factory CelestialBody.fromJson(Map<String, dynamic> json) =>
      _$CelestialBodyFromJson(json);
}

enum CelestialBodyId {
  sun, moon, mercury, venus, mars, jupiter, saturn, uranus, neptune, pluto,
  northNode, southNode, chiron, lilith,
}

@freezed
class Aspect with _$Aspect {
  const factory Aspect({
    required CelestialBodyId target,
    required AspectType type,                 // conjunction, sextile, square, trine, opposition
    required double orb,                      // [°] odchylenie od dokładnego aspektu
    required bool isApplying,                 // czy aspekt się zacieśnia
  }) = _Aspect;

  factory Aspect.fromJson(Map<String, dynamic> json) => _$AspectFromJson(json);
}

@freezed
class GeoLocation with _$GeoLocation {
  const factory GeoLocation({
    required double latitude,
    required double longitude,
    @Default(0.0) double elevationMeters,
  }) = _GeoLocation;

  factory GeoLocation.fromJson(Map<String, dynamic> json) =>
      _$GeoLocationFromJson(json);
}

enum ZodiacSign {
  aries, taurus, gemini, cancer, leo, virgo,
  libra, scorpio, sagittarius, capricorn, aquarius, pisces;

  /// Zwraca znak na podstawie długości ekliptycznej.
  /// Każdy znak zajmuje 30°: Baran 0-30°, Byk 30-60°, ...
  static ZodiacSign fromEclipticLongitude(double lon) {
    final normalized = lon % 360;
    return ZodiacSign.values[(normalized ~/ 30).toInt()];
  }
}

enum AstrologicalHouse { first, second, /* ... */ twelfth }
enum AspectType { conjunction, sextile, square, trine, opposition }
```

Klucz: Pola astronomiczne i astrologiczne są w jednej klasie, ale liczone z różnych układów współrzędnych. EphemerisService (SwissEph) zwraca eclipticLongitude natywnie; astronomy_engine_dart daje precyzyjne RA/Dec + alt/az dla obserwatora. CelestialCalculator to fasada która orkiestruje oba i składa CelestialBody.

### Night Mode (core/theme/night_mode_overlay.dart)

```dart
class NightModeOverlay extends ConsumerWidget {
  final Widget child;
  const NightModeOverlay({super.key, required this.child});

  @override
  Widget build(BuildContext context, WidgetRef ref) {
    final mode = ref.watch(themeModeProvider);
    if (mode != AppThemeMode.nightRed) return child;

    // ColorMatrix: usuwa kanały zielony i niebieski → tylko czerwony.
    // Zachowuje strukturę UI, ale chroni adaptację oka do ciemności (~30 min recovery).
    return ColorFiltered(
      colorFilter: const ColorFilter.matrix([
        1, 0, 0, 0, 0,   // R = R
        0, 0, 0, 0, 0,   // G = 0
        0, 0, 0, 0, 0,   // B = 0
        0, 0, 0, 1, 0,   // A = A
      ]),
      child: child,
    );
  }
}
```

Owijamy całe drzewo MaterialApp.builder, więc filtr działa globalnie bez potrzeby ręcznego stylowania widgetów.

Dodatkowo: AppThemeMode.nightDark używa Color(0xFF000000) (true OLED black) zamiast szarości — oszczędność baterii na OLED.

## Auth & Security — design pod komercyjny release

Werdykt na pytanie "czy zabezpieczenia można dodać potem bez problemu": TAK, ale tylko jeśli teraz zrobimy 5 niżej wymienionych decyzji architektonicznych. Bez nich retrofit = migracja danych użytkowników + audyt = tygodnie pracy + ryzyko wycieku.

### MVP: co implementujemy teraz

Local profile + PIN/biometria (offline-first):

1. Onboarding flow — pierwsze uruchomienie: user wpisuje nick → wybiera PIN (4-6 cyfr) → opcjonalny biometric setup (local_auth plugin). Profil zapisany w Drift jako LocalProfile (UUID id, nick, pinHash, biometricEnabled, createdAt).
2. PIN hashing — PBKDF2-HMAC-SHA256 z 100k iteracji, sól per-profile (16 bajtów random), zapisany w flutter_secure_storage. Nie własna kryptografia — tylko crypto package.
3. Lock screen — po cold start app wymaga unlock (PIN lub biometria). Auto-lock po X minutach w tle (default 5, konfigurowalne).
4. Wszystkie wrażliwe dane w flutter_secure_storage — pinHash, salt, sessionToken (gdy dojdzie sync), API keys jeśli kiedyś jakieś będą. NIGDY shared_preferences ani plain Drift dla wrażliwych pól.
5. Routing guards — go_router redirect: jeśli authState != Authenticated → /lock. Z wyjątkiem /onboarding.

### 5 decyzji architektonicznych "now" które ratują "later"

| # | Decyzja teraz                                                                                  | Bez tego problem potem |
|---|------------------------------------------------------------------------------------------------|------------------------|
| 1 | AuthProvider interface (sealed/abstract) — wszyscy konsumenci uzależnieni od interfejsu, nie impl | Bez tego: każdy widget/repo z hardcoded LocalAuth wywołaniami → setki miejsc do refaktoru |
| 2 | UUID userId w wszystkich user-facing tabelach (już w planie sync-ready)                        | Bez tego: migracja autoincrement → UUID na żywej bazie = ryzyko utraty danych |
| 3 | flutter_secure_storage dla wszystkiego wrażliwego od dnia 1                                    | Bez tego: jeśli choć raz pinHash trafi do shared_preferences, audyt security flagujący jako leak. Migracja istniejących userów = wymuszone re-setup PIN |
| 4 | PII filter przed Sentry/logami (pii_filter.dart) — usuwa email, lat/lon (zaokrągla do 0.1°), nick z error reportów | Bez tego: GDPR violation gdy crash report zawiera lokalizację user'a. Audyt = panika |
| 5 | DataPurger od MVP — funkcja "Delete all my data" w settings, hard delete z Drift + secure storage wipe + Sentry user opt-out | Bez tego: GDPR Right to Erasure niezgodność. Implementacja w pożarze przed release = błędy = wycieki |

### Co MOŻE czekać do komercyjnego release (bez bólu)

- Email verification flow — dochodzi z Supabase Auth, LocalAuthProvider go nie ma (i nie potrzebuje)
- Password reset — to samo
- 2FA (TOTP) — dodawalne jako opt-in feature later
- Session management UI ("active devices") — feature dochodzi z syncem
- Privacy policy / Terms of Service — TEKST może czekać, ale stuby w settings + linki muszą być od MVP (Apple Store wymaga)
- Rate limiting / brute-force protection — server-side, dochodzi z Supabase
- OAuth (Google/Apple Sign-In) — pluggable jako kolejna AuthProvider impl
- Audit logs (kto kiedy zalogował) — server-side z Supabase, MVP nie potrzebuje

### Threat model MVP (offline-first, pre-cloud)

| Threat                                                          | Mitigation w MVP |
|-----------------------------------------------------------------|------------------|
| Telefon ukradziony, atakujący chce dostęp do danych             | PIN (PBKDF2 100k iter, 6-cyfrowy = ~1M kombinacji + lockout po 5 nieudanych) + biometria opcjonalna. Drift database NOT encrypted w MVP — można dodać sqlcipher later (drift wspiera). |
| Apka loguje crash → Sentry leakuje lokalizację                  | pii_filter.dart przed Sentry.beforeSend |
| User chce usunąć konto/dane                                     | DataPurger w settings (hard delete + secure storage wipe) |
| Migracja do cloud: jak nie złamać UX?                           | Lokalny userId (UUID) staje się Supabase user id po sign-up. Drift userId column już wypełniona. Zero migracji danych. |

Co NIE jest w MVP threat modelu (dochodzi z syncem):
- Network MitM (HTTPS pinning)
- Server-side data breach
- API rate limiting
- DDoS

## Rekomendacje "na później" — werdykt: każdą da się dodać potem bez problemu

Decyzja: nie implementujemy ich w MVP, ale teraz wbudowujemy architectural hooks tak żeby przyszła implementacja była szybka i nie wymagała refaktoru.

| Feature                           | Czy odłożenie blokuje?                | Co robimy teraz (hook) | Co robimy potem (impl) |
|-----------------------------------|---------------------------------------|------------------------|------------------------|
| Plate Solving offline             | NIE                                   | Pusty folder features/plate_solving/ + README. pubspec.yaml: zarezerwować dependency placeholder w komentarzu. | Dodać dependency (FFI binding do astrometry.net C lib lub tetra3 port), katalog Tycho-2 ściągany on-demand. |
| AR Sky Map                        | NIE                                   | Pusty folder features/ar_sky_map/ + README. Camera permission w AndroidManifest.xml + Info.plist już w MVP (sama prośba dopiero gdy feature użyte — to standard, nie blokuje store review). | Dodać arcore_flutter_plugin + arkit_plugin, sensor fusion, render. |
| Eksport Stellarium/SkySafari      | NIE                                   | Pusty folder features/exporters/ + README. | Klasa per format (StellariumExporter, SkySafariExporter) + integracja z share_plus. Trywialne. |
| Sync (Supabase)                   | NIE — pod warunkiem sync-ready schemy | Drift schema MVP-ready dla synca: UUID PK, createdAt/updatedAt/deletedAt, kolumna userId (NULLABLE). Detale niżej. | Supabase Auth + Realtime + push/pull queue + conflict resolution. Wykorzysta istniejącą schemę bez migracji. |

Kluczowa obserwacja: wszystkie 4 features są izolowane od reszty (osobne moduły, nie zmieniają core/data). Sync to jedyny przypadek gdzie odłożenie byłoby problemem gdyby schema była naiwna — ale z poniższymi decyzjami nie jest.

### Sync-ready Drift schema (zero kosztu w MVP)

W tabelach user-facing (session_logs, equipment, natal_charts):

1. UUID v4 jako PK zamiast autoIncrement(). Drift: TextColumn get id => text().clientDefault(() => Uuid().v4())();
2. Audit columns (NOT NULL createdAt, updatedAt; NULLABLE deletedAt):
   - DAO helper _touch() ustawia updatedAt = DateTime.now().toUtc() przy każdym updatecie.
3. Soft delete — _softDelete() ustawia deletedAt, queries filtrują WHERE deletedAt IS NULL.
4. Reservation column userId (TEXT NULLABLE) — w MVP zawsze NULL, w sync mode wypełniana z Supabase Auth.

To kosztuje ~30 min w MVP. Bez tego za 6 miesięcy → migracja UUIDs na żywej bazie + risk utraty danych.
